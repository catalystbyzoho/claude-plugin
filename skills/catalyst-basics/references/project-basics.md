# Catalyst Project Basics — Reference

## Project Constraints (Know Before Creating)

- **First project must be created from the console** — `catalyst init --org <orgId> -p <projectId> -ni` can link subsequent projects from the CLI, but the very first one must be created at https://console.catalyst.zoho.com.
- **Project name rules** — alphanumeric, underscores (`_`), and hyphens (`-`) only. No spaces or special characters.
- **50-project limit** per account — contact support@zohocatalyst.com to request an increase.
- **Console URL differs by data center:**
  - US and others: `https://console.catalyst.zoho.com/baas/index`
  - EU: `https://console.catalyst.zoho.eu/baas/index`
  - IN: `https://console.catalyst.zoho.in/baas/index`

### "Start Exploring" Requirement

Before any service can be deployed to Production, a user **must click "Start Exploring"** for that service in the Catalyst console. This is a one-time activation per service per project — without it, deploying that service to Production is blocked. This applies to all services: Slate, Functions, AppSail, Data Store, etc.

## Project Directory Structure

When `catalyst init --org <orgId> -p <projectId> -ni` is run, a standard project layout is created:

```
my-catalyst-project/
├── catalyst.json                 # Auto-generated project config — NEVER create manually
├── .catalystrc                   # Auto-generated project identity — NEVER create manually
├── functions/                    # All server-side functions
│   └── function_name/
│       ├── index.js              # Entry point (Node.js)
│       ├── catalyst-config.json  # Function-specific config
│       ├── package.json
│       └── node_modules/
├── my-slate-app/                 # Slate frontend (PREFERRED for new projects)
│   └── catalyst-config.json
├── client/                       # ⚠️ LEGACY — use Slate instead
└── appsail/                      # AppSail services (optional)
    ├── app.js
    ├── catalyst-config.json
    └── package.json
```

Key rules:
- The `functions/` directory name is fixed and cannot be renamed.
- Each function must be in its own subdirectory under `functions/`.
- **For frontends, always use Slate** — do NOT enable the legacy "Client" feature during `catalyst init` (i.e. never pass it among the init features).
- `catalyst.json` and `.catalystrc` are auto-generated — **NEVER create them manually**.

## catalyst.json

Holds deployment configuration. Auto-generated by the CLI.

```json
{
  "functions": {
    "targets": ["myFunction1", "myFunction2"],
    "ignore": [],
    "source": "functions"
  },
  "appsail": {
    "targets": ["my-appsail-service"],
    "source": "appsail"
  },
  "slate": [
    { "name": "my-frontend", "source": "/absolute/path/to/client" }
  ]
}
```

- The `functions` block **must** include `targets`, `ignore`, and `source` or CLI errors.
- Slate `source` **must be an absolute path**.

## .catalystrc — Project Identity File

CLI-generated when you run `catalyst init … -ni`. **Never author or edit it by hand.** If missing, `catalyst deploy -ni` fails.

```json
{
  "defaults": { "project": 1, "env": 1 },
  "actives":  { "project": 1, "env": 1 },
  "projects": [
    {
      "idx": 1,
      "id": "21624000000082001",
      "name": "Acommerce",
      "domain": { "id": "50042231120", "name": "acommerce-60047217571.development" },
      "timezone": "Asia/Kolkata",
      "env": [
        { "idx": 1, "id": "60047217571", "name": "Development", "type": 3 }
      ]
    }
  ]
}
```

This file already contains everything the AI needs to know **which org and project it is working in** — no MCP call or console lookup required to establish context:

| Value | Path in `.catalystrc` | Notes |
|-------|-----------------------|-------|
| **Org ID** | `projects[].env[].id` | Catalyst stores the org identifier on the environment entry (e.g. `60047217571`). |
| **Project ID** | `projects[].id` | e.g. `21624000000082001`. |
| **Environment** | `projects[].env[].name` | `Development` or `Production`. |
| **Domain** | `projects[].domain.name` | e.g. `acommerce-60047217571.development`. |
| **Timezone** | `projects[].timezone` | IANA zone. |

Use `actives.project` / `actives.env` (1-based `idx`) to pick the currently selected project and environment from the arrays; fall back to the first entry.

> **Establishing context from this file is governed by the canonical pre-flight:** `preflight.md`. When `.catalystrc` is present, read the org/project from it and confirm the MCP account and CLI agree (via `CatalystbyZoho_Get_Project_By_Id`) before operating — rather than re-deriving context with `List_All_Organizations` / `List_All_Projects`.

## catalyst-config.json for Functions

```json
{
  "deployment": {
    "name": "my_function",
    "type": "advancedio",
    "stack": "node20",
    "env_variables": {}
  },
  "execution": {
    "main": "index.js"
  }
}
```

Valid `type` values (use exactly as shown — do not change this after function creation):

| Function Type | `type` value |
|--------------|-------------|
| Basic I/O | `basicio` |
| Advanced I/O | `advancedio` |
| Cron | `cron` |
| Job | `job` |
| Event | `event` |
| Integration | `integration` |
| Browser Logic | `browserlogic` |

> `browserlogic` — NOT `browselogic`.

Valid `stack` values: `node24` *(recommended)*, `node22`, `node20`, `node18`, `node16`, `node14`, `node12`, `java25`, `java21`, `java17`, `java11`, `java8`, `python_3_13`, `python_3_12`, `python_3_11`, `python_3_10`

## Environments

> **Canonical reference.** This is the single source of truth for the three Catalyst
> environments and the local-first development loop. Other skills link here instead of
> restating it — if the model changes, change it *here only*.

Catalyst has **three** environments:

| # | Environment | Where it runs | What it's for |
|---|-------------|---------------|---------------|
| 1 | **Local** | The developer's own machine (`catalyst serve`) | Developing and testing the components you run yourself — **Functions, AppSail, Slate** — before anything ships. |
| 2 | **Development** | Standalone remote sandbox | Where local code is shipped and tested against real remote services. Free within limits. `catalyst deploy` targets this by default. |
| 3 | **Production** | Standalone remote, stable | The live, production-ready application. Requires billing. Resources are **not created here directly** — changes reach Production only by migrating them up from Development. |

### Local ↔ Development are coupled

Local is **not** a fully self-contained environment — it is coupled to Development:

- **Served locally** (run on your machine): `catalyst serve` runs your **Functions, AppSail, and Slate** code on the dev machine with hot reload. This is where you iterate and test.
- **Proxied to Development**: calls to managed backend features that have no local runtime — **Data Store, Stratus, NoSQL, Cache, Authentication, Zia, Signals, Job Scheduling**, etc. — are proxied from your local session to the **Development** environment. Their data lives in Development.

The practical consequence: **Local cannot work to its full scope without a Development environment.** A `catalyst serve` session reads and writes real Development data through the proxy. Treat local Data Store / Stratus operations as operations on Development data, not a throwaway sandbox.

### Development → Production is a migration, not a deploy

You do not build in Production. You promote a validated Development setup up to Production (see the **Dev-to-Prod promotion checklist** below). `catalyst deploy --production` and console-based deploys carry the migrated changes up; direct resource creation in Production is not supported.

### The local-first development loop

For any component you run yourself (Functions, AppSail, Slate), **always follow this loop before deploying to Development** — do not deploy untested code straight to the remote:

1. **Build** — write/scaffold the component (`functions:add`, `appsail:add`, `slate:create`).
2. **Serve locally** — `catalyst serve` (port is dynamic; never hardcode). Managed-service calls proxy to Development automatically.
3. **Test locally** — exercise the component against the local server:
   - Functions: `curl` the local `/server/<fn>/execute` URL, or `catalyst functions:execute <fn> --input …` for Event/Cron/Job/Integration functions.
   - AppSail: hit the local server's routes.
   - Slate: open the local URL the CLI prints and click through the UI.
   - Run any unit/integration tests the project defines (`npm test`, `pytest`, etc.) at this stage.
4. **Fix and re-serve** — iterate locally until it works. Local iteration is free and fast; a failed remote deploy is slow and pollutes Development.
5. **Deploy to Development** — only once it works locally: `catalyst deploy -ni` (or `--only <component>`).
6. **Verify in Development** — smoke-test the deployed component on the Development URL.
7. **Promote to Production** — only after Development is verified, migrate up (checklist below).

> **Default rule for agents:** never run `catalyst deploy` for freshly written or changed serve-able code without first serving it locally and testing it. If the user asks to "deploy", serve + test locally first, report the result, then deploy — unless the user explicitly says to skip local testing.

### Application URL Format

| Environment | URL Pattern | Example |
|-------------|-------------|----------|
| Development | `https://{project-domain}.development.catalystserverless.com` | `https://shipmenttracking-57673975.development.catalystserverless.com` |
| Production | `https://{project-domain}.catalystserverless.com` | `https://shipmenttracking-57673975.catalystserverless.com` |

The project domain is auto-generated when you first host a web client.

### Dev-to-Prod promotion checklist

This is the final leg of the local-first loop — it assumes the code was already served + tested on Local and is now working in Development. Production changes are **migrated up** from Development, never authored directly.

1. **Verify in Development** — all functions, AppSail, and frontend working correctly (after passing local serve + test).
2. **Set up billing** — Catalyst Console → Settings → Billing.
3. **Deploy to Production** — Console → Deploy → select Production environment, or `catalyst deploy --production` from the CLI (double-check the project context first).
4. **Update environment variables** — Production uses separate env vars.
5. **Swap ZAIDs** — Production has different ZAIDs than Development. Use env variables, never hardcode.
6. **Reconfigure social login** — OAuth redirect URLs must point to the Production domain.
7. **Map custom domain** — Console → Domain Mapping → add your production domain.
8. **Test the full flow** — auth, data, file uploads, email — all on the Production URL.
9. **Monitor** — enable APM and Application Alerts for Production.

## Catalyst IDs Quick Reference

| ID | What | Where to Find | Format |
|---|---|---|---|
| **Project ID** | Project identifier | `.catalystrc` (`projects[].id`); Settings → General; `catalyst project:list` | Numeric string |
| **ZAID** | App-to-environment mapping | Settings → Environments → General tab | Numeric (differs dev/prod!) |
| **API Key** | API Gateway auth key | Settings → Environments → General tab | String (common in Dev across all projects; unique per project in Prod) |
| **Table ID** | Data Store table | Cloud Scale → Data Store → click table | Numeric |
| **ROWID** | Data Store row | Auto-assigned; returned in queries | BigInt |
| **Segment ID** | Cache segment | Cloud Scale → Cache → segment list | Numeric |
| **Function ID** | Serverless function | Serverless → Functions → function details | Numeric |
| **Circuit ID** | Circuits workflow | Serverless → Circuits → circuit details | Numeric |
| **Pool ID** | Job Scheduling pool | Job Scheduling → pool details | Numeric |
| **Bot ID** | ConvoKraft bot | ConvoKraft → bot details | String |
| **ZUID** | Zoho user (per-app) | Auth API responses; Authentication → Users | Numeric string |
| **User ID** | Catalyst-only user | Auth API responses; Authentication → Users | Numeric |
| **Org ID / ZAAID** | Organization | `.catalystrc` (`projects[].env[].id`); Auth API responses; Authentication → Users | Numeric string |
| **Bucket Name** | Stratus bucket | Cloud Scale → Stratus | String |
| **Collection Name** | NoSQL collection | Cloud Scale → NoSQL | String |

### ZAID (Critical)

- **Different in Development and Production.** This is the #1 source of auth issues when deploying.
- **Where:** Console → Settings → Environments → General tab.
- **Always pass as a string** (JavaScript loses precision on large integers).

```javascript
const signupConfig = {
  platform_type: 'web',
  zaid: '10014774358'  // ← Get from Settings → Environments (always a string)
};
```

### Table / Column IDs

```javascript
// Access by name (case-sensitive — must match console exactly)
const table = catalystApp.datastore().table('Employees');

// Access by ID (from Console → Data Store → click table)
const table = catalystApp.datastore().table(1510000000110121);
```

### Segment ID (Cache)

```javascript
// Segment ID from Console → Cache → segment list
const segment = catalystApp.cache().segment(SEGMENT_ID);
```

### Pool ID (Job Scheduling)

```javascript
// Pool ID from Console → Job Scheduling → pool details
const pool = catalystApp.jobScheduling().pool(POOL_ID);
```

### Circuit ID

```javascript
// Circuit ID from Console → Circuits → circuit details
const circuit = catalystApp.circuit();
const result = await circuit.execute(CIRCUIT_ID, { inputKey: 'value' });
```

## Catalyst Organizations

Catalyst supports multiple organizations per account under the same email address.

### Key concepts

- **Org owner**: Created the org. Can add collaborators (admins or project members), grant permissions per org or per project.
- **Collaborator types**: `Admin` (org-wide access) or `Project Member` (scoped to specific projects). Permissions are based on the assigned profile.
- **Org ID**: Auto-generated unique ID for each org. Part of the console URL: `https://console.catalyst.zoho.com/baas/{OrgID}/index`
- **Default org**: The org Catalyst assigns you to at signup. Can be changed to any other org you own.

### Default organization behavior — critical for MCP and API usage

The default org affects everything:
- **CLI login** — always logs into the default org unless you switch explicitly.
- **Every API call** — executed against a project in the default org unless you pass the `CATALYST-ORG: {org_id}` header explicitly.
- **MCP tools** — `CatalystbyZoho_*` calls target the default org. When `.catalystrc` is present, the correct org ID (`projects[].env[].id`) is already known — pass it explicitly in subsequent calls. Only when `.catalystrc` is absent, call `List_All_Organizations` first to confirm the org.

> **Cannot delete the default org.** Set a different org as default before attempting to delete.

### Accessing the multi-org organization

Console → profile icon (top-left) → Organizations dropdown → **Manage Organizations**

The organization view shows each org's unique ID and console URL.

### Switching organizations / projects via CLI

```bash
catalyst whoami                                        # shows current logged-in user
catalyst project:use <name-or-id> -ni                  # set the active project
catalyst project:use <name-or-id> --org <org-id> -ni   # target a project in a different org
```

> There is no `catalyst switch:org` command. Use `catalyst project:use` (optionally with `--org`) to change the active org/project. `catalyst project:use` also rewrites `.catalystrc`.

## IaC Export / Import

Catalyst supports project export and import (Infrastructure as Code) for migrating between data centers, duplicating projects, or sharing project templates.

### What is exported

- ✅ Component configurations (Data Store schema, Cache segments, Circuits, Security Rules, API Gateway routes, email templates, user profiles, etc.)
- ✅ Function and client code
- ❌ **Data is NOT exported** — no DataStore rows, no file contents, no user list records

### How to export

Console → Settings → General → **Export Project** → downloads a ZIP file in Catalyst IaC format.

Also available via CLI (the export/import commands live under the `iac:` namespace — see `cli.md` for flags):
```bash
catalyst iac:export           # export the project's config + code
catalyst iac:pack             # generate an import-ready ZIP without exporting first
```

### How to import

Console → index page → **Import Project** → upload the ZIP → new project is created.

Also available via CLI:
```bash
catalyst iac:import --file ./my-project.zip
```

### Common use cases

| Use case | How |
|----------|-----|
| Move project to a different DC (e.g., US → EU) | Export → log into EU console → Import |
| Duplicate a project for a new client | Export → Import as new project in same or different org |
| Test a GitHub-hosted project | Clone repo → generate ZIP → import and test |
| CI/CD project templating | Script `catalyst iac:export` + `catalyst iac:import` |

## Common Errors

| Error | Cause | Fix |
|-------|-------|-----|
| `catalyst init` fails with "project already exists" | Re-running init in a directory that already has `catalyst.json` | Delete `catalyst.json` and `.catalystrc` and re-run `catalyst init --org <orgId> -p <projectId> -ni`, or run it in a fresh directory |
| `No project found` when running `catalyst deploy` | Working directory has no `catalyst.json` | Run `catalyst init --org <orgId> -p <projectId> -ni` first, or `cd` to the correct project root |
| Wrong environment targeted on deploy | Catalyst selects the environment at deploy time — there is no environment-switch command | Use `catalyst deploy --production -ni` to target Production; omit `--production` for Development |
| `Permission denied` on deploy | API key lacks deploy permissions for this project | Confirm the key has Developer or higher role in Console → Project Settings |
